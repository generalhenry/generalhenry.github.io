We believe code samples should be runnable.  However code doesn't run in a vacuum, it needs an environment.  Code samples at Runnable.com, are both the code and the environment to run that code.

Every time you visit a project page (such as hello-world-in-express-for-node-js) we look up the Docker image we have associated with it and create a container out of that image.  The Docker container provides the code sample with the environment it needs.  Then to bring the code and environment to life in your browser.

Docker enables us to give full access to a full linux instance to each user.  Docker is a tool set build out of two basic components, a union file system, and lxc, linux containers.  The union file system allows us to have limitless numbers of projects, the containers give us the sandboxing we need to safely give users access and Docker makes the whole thing manageable.

Once Docker has set up a container we need to finish up preparing the environment and expose it to the browser.  At the core of how we do is a project called dockworker.  Dockworker starts services, exposes a terminal (the terminal is an iframe direct to the container), exposes the filesystem, and most importantly exposes the build/run process of the project.  In short dockworker gives you the access you would expect to have locally in the browser.

Finally we bring it all together with a series of routers and proxies.  Each project page visit finds an dock, creates a container and routes two subdomains to that container, one for the terminal/file access and another for the projects web output.  Then to be able to start containers on demand, allowing users to use containers as drafts, we have a secondary routing proxy that starts the container if it's not started.

Runnable.com runs any code example that will run on linux and has web and/or terminal output.  But that's just the start.  We want all code to be runnable.
